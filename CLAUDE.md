# CLAUDE.md

## 目的与非目标

本文件定义仓库级 AI 优先工程规范。
它用于在快速 AI 协作开发中保持架构质量、设计质量与代码质量。

本文件不定义业务需求、产品范围、路线图优先级或功能策略。

## AI 开发契约

所有 AI 贡献者 MUST 遵循以下规则：

1. 先复用：新增实现前先搜索现有实现。
2. 当现有结构明显薄弱时，优先做能改善设计的改动，而不是局部补丁。
3. 保持改动显式、可评审，并提供清晰理由与验证证据。
4. 避免隐藏副作用与无关改动。
5. 交付时必须使用本文件中的完成报告模板。

## 注释、作者与版权规范（强制）

1. 注释必须解释“设计意图/约束/边界”，而不只复述代码字面行为。
2. 核心契约文件（如 `interfaces.go`）MUST 提供包级注释与接口级注释，明确职责、依赖方向与迁移背景。
3. 对外可见 API（导出类型、接口、函数、常量）SHOULD 提供可审阅的文档注释；涉及状态机、权限、事件语义的导出符号 MUST 注释。
4. 禁止无信息量注释（如“给变量赋值”）；禁止以注释替代实现（TODO 占位不可作为交付结果）。
5. 注释必须与实现同步更新；修改行为而未更新注释视为不完整交付。
6. 新增源代码文件（非测试夹具、非第三方镜像）MUST 使用统一文件头信息，推荐格式：
   - `Copyright (c) <year> <owner>`
   - `Author: <author-or-team>`
   - `SPDX-License-Identifier: <id>`
7. 作者信息必须可追踪且稳定：优先使用团队/仓库统一身份；不得伪造、冒用或写入无责任归属的昵称。
8. 版权与许可标识必须与仓库 License 策略一致；禁止引入与仓库不兼容的许可证声明。
9. 对第三方/复制代码 MUST 保留原始版权与许可证声明，并按其许可证要求补充归属信息。
10. 自动生成文件应使用生成标记（如 `Code generated ... DO NOT EDIT.`）；作者与版权遵循生成器模板，不手工随意改写。

## 架构不变式（不得破坏）

1. 控制权保持 Hub 中心化：Desktop/Mobile 客户端不得绕过 Hub 执行控制。
2. 执行语义保持一致：单会话同一时刻仅一个活跃执行，且会话内 FIFO。
3. 工作区边界必须持续生效，用于授权与数据隔离。
4. 共享契约变更必须联动：Hub API、OpenAPI、共享 TypeScript API 模型必须保持一致。

## Agent v4 前置治理（文档双轨）

1. 项目指令文档采用双轨等量维护：`AGENTS.md` 与 `CLAUDE.md`。
2. 同目录下两份文档正文必须一致（允许标题行不同）。
3. 指令发现优先级以实现为准：`AGENTS.override.md` -> `AGENTS.md` -> `CLAUDE.md`。
4. 每个目录最多生效一个指令文件（按上述优先级选择）。
5. Agent v4 重构阶段，`services/hub/internal/agent/` 及其子包必须具备成对指令文档。
6. 若更新了任一轨道文档，必须在同次提交中同步更新另一轨道。

## Change Surface Lock（仅针对既有文件）

为保证 AI 改动可控，必须启用 Change Surface Lock。

1. 开始实现前，AI MUST 声明本任务的 `Locked existing files`。
2. AI 只能修改锁定列表中的既有文件。
3. 新文件可直接新增，无需预先列出。
4. 若必须修改未锁定既有文件，AI MUST 先更新锁定列表并说明原因后再修改。
5. 删除或重命名既有文件视为修改既有文件，必须预锁定。
6. 最终输出 MUST 包含 `Locked existing files`。
7. 最终输出 MUST 包含 `Actually modified existing files`。
8. 最终输出 MUST 包含 `Lock delta`，明确是否超出范围。

## 前端工程标准

1. 新增实现前优先复用已有组件、store、design token 与交互模式。
2. 如引入新 UI 原语，AI MUST 说明为何现有组件不适用。
3. 新增或重构 Vue 代码默认使用 Vue 3 Composition API 与 `<script setup lang="ts">`，除非文件明确为遗留模式。
4. 保持视图逻辑、状态逻辑、API 集成职责分离。
5. 优先使用已有设计 token 与共享样式，避免一次性 CSS 值。
6. 行为变更时补充或更新测试。
7. 避免复制粘贴式组件变体，应抽取共享逻辑。

## 后端工程标准

1. 新增工具前优先复用已有 helper、validator、adapter 与错误处理模式。
2. 保持 HTTP handler 轻量，将业务/领域行为下沉到可复用内部模块。
3. 权限检查与安全检查必须集中且一致。
4. 重构时保持稳定错误语义与响应模型。
5. 行为复杂或边界较多逻辑优先采用表驱动测试。
6. 避免在多个包重复解析、校验、存储或传输逻辑。

## 重构策略（开发阶段）

项目处于活跃开发阶段。只要能提升系统质量，允许并鼓励收益导向的破坏性重构。

当 AI 发现以下情况时 SHOULD 触发“重构优先”行为：

1. 存在应被共享的重复逻辑。
2. 存在分层违例或紧耦合。
3. 命名不佳、模块边界不清、职责混杂。
4. 临时补丁会增加技术债。

发生破坏性或结构性重构时 MUST 提供：

1. 正在消除的设计债。
2. 新结构更优的原因。
3. 受影响文件与兼容性影响。
4. 证明关键行为无回归的验证证据。

## Anti-Laziness 规则（AI）

AI MUST NOT：

1. 在会恶化架构时默认做最小改动。
2. 放弃复用而复制已有逻辑。
3. 以 TODO 占位实现作为最终输出。
4. 对行为变更跳过测试或验证。
5. 无说明地引入跨模块隐式耦合。

AI MUST：

1. 优先保证整体设计一致性，而非补丁堆叠。
2. 当重构替代旧路径时清理死代码。
3. 命名精准，并与模块职责一致。
4. 在完成报告中显式说明关键取舍。

## 验证策略（分层门禁）

默认执行与改动范围相匹配的最小充分验证。
对高风险或跨栈改动升级验证。

1. Desktop 改动：运行 `pnpm lint` 与 `pnpm test`。
2. Desktop 高风险改动（执行流/状态编排/运行时关键 UI）：额外运行 `pnpm test:strict` 与 `pnpm e2e:smoke`。
3. Mobile 改动：运行 `pnpm lint:mobile` 与 `pnpm test:mobile`。
4. Mobile 高风险改动：额外运行 `pnpm build:mobile` 与 `pnpm --filter @goyais/mobile e2e:smoke`。
5. Hub 改动：运行 `cd services/hub && go test ./... && go vet ./...`。
6. 仅文档改动：运行 `pnpm docs:build`；若 slides 变更，再运行 `pnpm slides:build`。
7. 跨栈或发布敏感改动：额外运行 `make health`。

每次交付都必须列出“实际运行的命令”及其结果。

## 仓库内指令优先级

指令发现遵循 `projectdocs` 已实现行为：

1. 从 git 根目录遍历到当前工作目录。
2. 每个目录最多加载一个指令文件，优先级为：
`AGENTS.override.md` -> `AGENTS.md` -> `CLAUDE.md`。
3. 按根到叶顺序拼接选中的文件。
4. 总指令大小受 `GOYAIS_PROJECT_DOC_MAX_BYTES` 限制（默认 32768 字节），超限会截断。

## 本文件维护规则

1. 规则应具体、技术化、可执行。
2. 文件应保持精简，避免在指令加载时被截断。
3. 脚本或工具链变化后应同步更新命令示例。
4. 可执行规则优先使用 `MUST`/`SHOULD`。
5. 不在此文件重复产品/业务策略。
