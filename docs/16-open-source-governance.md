# Goyais 开源治理与企业级工程规范（GitHub / Apache 对标）

> 本文档定义 Goyais 作为 GitHub 开源项目的治理、协作、发布与安全基线，对标 Apache 顶级项目实践（社区治理、开放协作、可追溯发布、长期兼容）。

最后更新：2026-02-09

---

## 1. 目标与范围

1. 保证项目在开源协作场景下可持续演进。
2. 保证架构与接口变更可审计、可追溯、可回滚。
3. 保证社区贡献有明确流程、质量门禁与行为约束。
4. 保证版本发布具备企业级交付质量与供应链透明度。

---

## 2. 治理原则

1. 文档即契约：设计、接口、事件、策略必须有权威来源并保持一致。
2. 共识优先：重大变更通过 issue/PR 讨论并记录决策。
3. 向后兼容优先：对外契约优先增量扩展，避免破坏式演进。
4. 可验证交付：每次合并与发布都必须有可复现的验证证据。
5. 实现遵循设计：开发必须严格按设计文档执行；发现文档错误先修正文档。

---

以下文件必须存在并保持最新：

## 4. 角色与职责

### 4.1 Maintainer

1. 维护技术方向与发布节奏。
2. 审核高风险 PR（架构、接口、安全）。
3. 处理版本发布与应急修复决策。

### 4.2 Committer/Reviewer

1. 审核代码质量、测试完整性、兼容性风险。
2. 确保 PR 满足门禁并具备可追溯证据。

### 4.3 Contributor

1. 遵守贡献流程与行为准则。
2. 提供最小可复现实例与测试证明。

---

## 5. 贡献与评审标准

### 5.1 提交流程

1. 所有改动通过分支 + PR 进行。
2. PR 必须关联任务 ID 或 issue。
3. PR 描述必须包含：
   - 变更动机
   - 变更范围
   - 测试结果
   - 兼容性影响

### 5.2 评审门禁

1. 至少 1 名 Reviewer 通过（关键改动建议 2 名）。
2. 必须通过自动化检查：lint + test + build。
3. 关键链路变更必须附回归或压测结果。
4. 契约改动必须同步更新文档（00-11、12、15）。
5. 若 PR 涉及“文档错误纠正”，必须清晰说明修正原因和影响范围。

---

## 6. 分支与版本策略

1. 默认分支受保护，禁止直接 push。
2. 功能分支命名建议：`codex/<task-id>-<topic>`。
3. 版本建议采用 SemVer（MAJOR.MINOR.PATCH）：
   - MAJOR：破坏性变更
   - MINOR：向后兼容新增
   - PATCH：缺陷修复

---

## 7. 契约兼容性政策

1. 对外契约范围：REST API、SSE、错误码、事件类型。
2. 非兼容变更必须：
   - 提前发布弃用公告
   - 提供迁移文档
   - 保留兼容窗口
3. 所有契约改动必须在 `docs/12-dev-kickoff-package.md` 与对应权威文档中同步。

---

## 8. 安全与漏洞响应

1. 漏洞报告入口以 `SECURITY.md` 为准。
2. 安全问题优先私下披露，不公开利用细节。
3. 处理流程必须包含：
   - 受影响范围评估
   - 修复与回归验证
   - 发布说明与缓解建议

---

## 9. 发布工程标准

每次正式发布至少应包含：

1. Release Notes / CHANGELOG。
2. 版本标签与构建产物校验信息。
3. 依赖与供应链可见性材料（例如 SBOM）。
4. 升级/回滚说明。

发布前必须执行：

1. 全量回归（核心单测 + 集成 + 关键 e2e）。
2. 合规检查（许可证、依赖安全、敏感信息扫描）。
3. 文档一致性检查（契约、运行手册、发布说明）。
